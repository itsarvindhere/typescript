Let's take an example we had taken in the Branded Types section -

    declare const __brand: unique symbol;
    type Branded<T, TBrand> = T & { [__brand]: TBrand };

    type Valid<T> = Branded<T, 'Valid'>;

    interface PasswordValues {
        password: string;
        confirmPassword: string;
    }

    const isValidPassword = (values: PasswordValues) => {
        if (values.password !== values.confirmPassword) {
            return false;
        }
        return true;
    }

    const createUserOnApi = (values: Valid<PasswordValues>) => {
        // API call to create user
    }

    const onSubmitHandler = (values: PasswordValues) => {
        if (isValidPassword(values)) {
            createUserOnApi(values);
        }
    }

Here, we have a function 'isValidPassword' that checks if the password and confirm password fields match. If they do, it returns true. We also have a function 'createUserOnApi' that takes a 'Valid<PasswordValues>' as an argument, which means it expects a branded type that indicates that the password values are valid.

Right now, we will get an error because we are passing 'values' of type 'PasswordValues' to 'createUserOnApi', which expects a 'Valid<PasswordValues>'.

Well, here, we know that our 'isValidPassword' is a function that returns a boolean. If it returns true, then we can be sure that the 'values' are valid. So, we can update its return type as -    

    const isValidPassword = (values: PasswordValues): values is Valid<PasswordValues> => {
        if (values.password !== values.confirmPassword) {
            return false;
        }
        return true;
    }

Here, we are telling TypeScript that if 'isValidPassword' returns true, then the 'values' parameter is of type 'Valid<PasswordValues>'. This is a type predicate that allows us to narrow down the type of 'values' within the scope of the 'if' statement.

And the error goes away.
