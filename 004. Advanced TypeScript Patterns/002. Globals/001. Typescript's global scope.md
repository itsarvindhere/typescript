So, what even is 'Global Scope'?

In general programming terms, 'Global Scope' refers to the area of a program where certain variables, functions, or types are accessible from anywhere in the code.

In TypeScript, this means that any variable, function, or type declared in the global scope can be accessed and used throughout the entire codebase without needing to import or reference it.

In browsers, the global scope is the 'window' object. Variables declared with 'var' at the top level are attached to 'window'. In Node.js, the global scope is the 'global' object (or 'globalThis').

A .ts file without any 'import' or 'export' statements is treated as a script, and its top-level declarations are in the global scope.

A .ts file with 'import' or 'export' is treated as a module, and its top-level declarations are scoped to that module (not global).

So, let's say that we are attaching some functions or variables in the global scope like this -

    globalThis.myFunc = () => true;
    globalThis.myVar = 1;

Now, ideally, we should be able to directly access 'myFunc' and 'myVar' anywhere in our code without any issues, right?

    myFunc(); // ERROR - Cannot find name 'myFunc'.
    const x = myVar; // ERROR - Cannot find name 'myVar'.

But right now, if we try to access 'myFunc' or 'myVar' directly, TypeScript will throw an error because it doesn't know about these properties on the global scope even though they exist at runtime.

So, how do we tell TypeScript about these global properties?

That's where we can actually extend the global scope in TypeScript. We can declare a global augmentation to inform TypeScript about the new properties we've added to the global scope.

We do this using the 'declare global' syntax. Here's how we can do it:

    declare global {
        function myFunc(): boolean;
        var myVar: number;
    }

Here, we are saying to TypeScript that there is a function 'myFunc' that returns a boolean and a variable 'myVar' of type number in the global scope.

And now, TypeScript will recognize 'myFunc' and 'myVar' as valid global properties, and we can use them without any errors.

'declare' tells TypeScript: "trust me, this exists at runtime — don't generate any JavaScript for it, just note the type." It's purely a compile-time annotation. TypeScript believes you, but you're responsible for actually making it exist.

One thing worth mentioning is that you can only declare global augmentations in a module context. This means that the file where you declare the global augmentation must have at least one 'import' or 'export' statement to be treated as a module. If you try to declare global augmentations in a script context (a file without 'import' or 'export'), it will not work as expected.

So, even if you do not have any exported members in your file, you can still add an empty export statement to make it a module and allow global augmentations:

    export {}; // This makes the file a module

    declare global {
        function myFunc(): boolean;
        var myVar: number;
    }

Now, even though this is a really good way to inform TypeScript about global properties, it's important to use this pattern judiciously. Overusing global scope can lead to code that is harder to maintain and debug, as it can create unintended side effects and make it difficult to track where certain variables or functions are being used or modified. Always consider whether there are better ways to structure your code before resorting to global scope.
