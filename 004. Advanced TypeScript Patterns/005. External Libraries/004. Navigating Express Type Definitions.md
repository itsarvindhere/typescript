After 'lodash' example, let's now take a look at 'express' which is also a very popular library for building web applications in Node.js.

Again, 'express' does not comes with its own type definitions, so we need to install them using the following command -

    npm install --save-dev @types/express

Let's say we have this code -

    import express from 'express';

    const app = express();

    app.get('/user', (req, res) => {
        console.log('User endpoint hit');
    });

Now, at first glance, it looks quite simple. But express' types can get really complicated if you look at their type definitions. For example, if you ctrl + click on the 'get' method, you will find something like this -

    export interface IRouterMatcher<
        T,
        Method extends "all" | "get" | "post" | "put" | "delete" | "patch" | "options" | "head" = any,
    > {
        <
            Route extends string | RegExp,
            P = RouteParameters<Route>,
            ResBody = any,
            ReqBody = any,
            ReqQuery = ParsedQs,
            LocalsObj extends Record<string, any> = Record<string, any>,
        >(
            // (it's used as the default type parameter for P)
            path: Route,
            // (This generic is meant to be passed explicitly.)
            ...handlers: Array<RequestHandler<P, ResBody, ReqBody, ReqQuery, LocalsObj>>
        ): T;
        <
            Path extends string | RegExp,
            P = RouteParameters<Path>,
            ResBody = any,
            ReqBody = any,
            ReqQuery = ParsedQs,
            LocalsObj extends Record<string, any> = Record<string, any>,
        >(
            // (it's used as the default type parameter for P)
            path: Path,
            // (This generic is meant to be passed explicitly.)
            ...handlers: Array<RequestHandlerParams<P, ResBody, ReqBody, ReqQuery, LocalsObj>>
        ): T;
        <
            P = ParamsDictionary,
            ResBody = any,
            ReqBody = any,
            ReqQuery = ParsedQs,
            LocalsObj extends Record<string, any> = Record<string, any>,
        >(
            path: PathParams,
            // (This generic is meant to be passed explicitly.)
            ...handlers: Array<RequestHandler<P, ResBody, ReqBody, ReqQuery, LocalsObj>>
        ): T;
        <
            P = ParamsDictionary,
            ResBody = any,
            ReqBody = any,
            ReqQuery = ParsedQs,
            LocalsObj extends Record<string, any> = Record<string, any>,
        >(
            path: PathParams,
            // (This generic is meant to be passed explicitly.)
            ...handlers: Array<RequestHandlerParams<P, ResBody, ReqBody, ReqQuery, LocalsObj>>
        ): T;
        (path: PathParams, subApplication: Application): T;
    }

Yep, this is way too confusing at first glance. And if you are wondering why did we get an interface when we clicked on the 'get' method, it's because 'get' is defined like this in types -

    get: ((name: string) => any) & IRouterMatcher<this>;

So, it is a function that also has the properties of the 'IRouterMatcher' interface.

Inside this interface, if you notice, we have different method signatures. This is actually a way to define function overloads in Typescript using interfaces.

And so, you will see that these overloads are generic in nature and they accept a lot of type parameters. For example, the first overload accepts 6 type parameters -

    <
        Route extends string | RegExp,
        P = RouteParameters<Route>,
        ResBody = any,
        ReqBody = any,
        ReqQuery = ParsedQs,
        LocalsObj extends Record<string, any> = Record<string, any>,
    >(
        // (it's used as the default type parameter for P)
        path: Route,
        // (This generic is meant to be passed explicitly.)
        ...handlers: Array<RequestHandler<P, ResBody, ReqBody, ReqQuery, LocalsObj>>
    ): T;

So, even though the 'get' method looks simple, it is actually a very complex function with a lot of type parameters and overloads. This is one of the reasons why using external libraries can be quite challenging in Typescript, especially if the library has complex types.

The second argument to the 'get' method is an array of request handlers. And if you look at the type definition of 'RequestHandler', you will find that it is also a generic type with a lot of type parameters -

    export interface RequestHandler<
        P = ParamsDictionary,
        ResBody = any,
        ReqBody = any,
        ReqQuery = ParsedQs,
        LocalsObj extends Record<string, any> = Record<string, any>,
    > {
        // tslint:disable-next-line callable-types (This is extended from and can't extend from a type alias in ts<2.2)
        (
            req: Request<P, ResBody, ReqBody, ReqQuery, LocalsObj>,
            res: Response<ResBody, LocalsObj>,
            next: NextFunction,
        ): unknown;
    }

But this one is fairly simpler than the 'get' method. It is just a function that takes in a request, response and next function as arguments and returns an unknown type.

So, we can also write our code like this -

    const getUser: RequestHandler =  (req, res) => {
        console.log('User endpoint hit');
    };

    app.get('/user', getUser);

This works in the same way as before but this time, we have explicitly defined the type of the 'getUser' function as 'RequestHandler'.

Now, what does this 'RequestHandler' take as type parameters? It takes 5 type parameters -

        P = core.ParamsDictionary,
        ResBody = any,
        ReqBody = any,
        ReqQuery = core.Query,
        Locals extends Record<string, any> = Record<string, any>

What is this 'P'? If we do not provide 'P', we can see that it defaults to 'core.ParamsDictionary'. And if we look at the type definition of 'ParamsDictionary', we will find that it is defined like this -

    export interface ParamsDictionary {
        [key: string]: string | string[];
        [key: number]: string;
    }

So, it is an object that can have any number of string or number keys and the values can be either a string or an array of strings.

Interesting. But, where is this 'P' even used inside the 'RequestHandler' type? If we look at the type definition of 'RequestHandler', we will find that it is used as the type of the 'req' parameter -

    (
        req: Request<P, ResBody, ReqBody, ReqQuery, LocalsObj>,
        res: Response<ResBody, LocalsObj>,
        next: NextFunction,
    ): unknown;

So, where is it used inside the 'Request' type?

We will see that 'Request' is typed like this -

    export interface Request<
        P = ParamsDictionary,
        ResBody = any,
        ReqBody = any,
        ReqQuery = ParsedQs,
        LocalsObj extends Record<string, any> = Record<string, any>,
    > extends http.IncomingMessage, Express.Request {...}

So, it extends the 'http.IncomingMessage' type and also has some additional properties defined in the 'Express.Request' interface. And this 'P' is actually the type of -

    params: P;

This basically means, if we have -

    const getUser: RequestHandler<{
        id: string
    }> =  (req, res) => {
        console.log('User endpoint hit');
    };

Then here, the 'P' type parameter is an object with an 'id' property of type string. And this means that the 'params' property of the 'req' object will have the type -

    {
        id: string
    }

This means, inside the function, we can actually access this 'id' param using -

    const getUser: RequestHandler<{
        id: string
    }> =  (req, res) => {
        console.log(req.params.id);
    };

This is beautiful. We have full type safety and autocompletion for the 'id' param inside our request handler function.

Let's take a look at another type parameter of the 'RequestHandler' type, which is 'ResBody'. This is the type of the response body. So, we can actually define what type of data we want to send in the response body and get full type safety for that as well. For example -

    const getUser: RequestHandler<{
        id: string
    }, {
        name: string
    }> =  (req, res) => {
        console.log('User endpoint hit', req.params.id);
        res.send({ name: 'John' });
    };
        
So here, we have defined the 'ResBody' type parameter as an object with a 'name' property of type string. This means that when we call the 'res.send' method, it will expect an object with a 'name' property of type string. If we try to send something else, we will get a type error.

Similarly, the third type parameter is actually 'ReqBody' which means what payload we are expecting in the request body. So, if we are expecting a payload with a 'name' property of type string, we can define our request handler like this -

    const getUser: RequestHandler<{}, {
        name: string
    }, {
        id: string
    }> =  (req, res) => {
        req.body.id
        console.log('User endpoint hit', req.body.id);
        res.send({ name: 'John' });
    };

So now, we are saying that we are not expecting any params in the URL (hence the empty object for the first type parameter), we are expecting a request body with an 'id' property of type string and we are sending a response body with a 'name' property of type string.

Finally, we can also define the type of the query parameters using the 'ReqQuery' type parameter. For example -

    const getUser: RequestHandler<{}, {
        name: string
    }, {
        id: string
    }, {
        search: string
    }> =  (req, res) => {
        req.body.id
        console.log('User endpoint hit', req.body.id);
        console.log("Query params", req.query.search);
        res.send({ name: 'John' });
    };

Now, we are saying that we are expecting a query parameter called 'search' of type string. So, when we access 'req.query.search', we will get full type safety and autocompletion for that as well.
