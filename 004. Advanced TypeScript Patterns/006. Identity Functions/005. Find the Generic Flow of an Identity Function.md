Alright, let's see a somewhat practical example of what we learned in the last section.

So, suppose, we have a config object -

    export const configObj = {
        routes: ["/", "/about", "/contact"],
        fetchers: {
            "/does-not-exist": () => {
                return {};
            }
        }
    };

This is a simple config object that has two properties - 'routes' and 'fetchers'. The 'routes' property is an array of strings, while the 'fetchers' property is an object where the keys are strings and the values are functions that return an object.

But what we ideally want is that each key in the 'fetchers' object should correspond to a route in the 'routes' array. In other words, we want to ensure that the keys in the 'fetchers' object are valid routes.

Right now, we can see that the key "/does-not-exist" in the 'fetchers' object does not correspond to any route in the 'routes' array. This is a problem because it can lead to runtime errors when we try to access a fetcher for a route that doesn't exist.

So, how do we enforce this relationship between the 'routes' and 'fetchers' properties in our config object?

We can create an identity function that takes in the config object and returns it, but with the added constraint that the keys in the 'fetchers' object must be valid routes.

    const makeConfigObj = <TRoute extends string>(obj: {
        routes: TRoute[],
        fetchers: Record<TRoute, () => any>
    }) => obj;

So, our identity function 'makeConfigObj' takes in an object that has a 'routes' property, which is an array of strings (TRoute[]), and a 'fetchers' property, which is an object where the keys are of type TRoute and the values are functions that return any type.

And since this is an identity function, it simply returns the object that it receives as an argument. All that changes here is the type and not the value.

Now, when we pass the config object to this function -  

    const config = makeConfigObj({
        routes: ["/", "/about", "/contact"],
        fetchers: {
            "/does-not-exist": () => { // ERROR
                return {};
            }
        }
    });

Now, we will start getting an error because if you hover over the 'config' variable, you will see that its type is -    

    const config: {
        routes: ("/" | "/about" | "/contact")[];
        fetchers: Record<"/" | "/about" | "/contact", () => any>;
    }

So, the 'fetchers' property is now of type 'Record<"/" | "/about" | "/contact", () => any>', which means that the keys in the 'fetchers' object must be one of the routes defined in the 'routes' array. That's why now, we are getting an error for the key "/does-not-exist" because it is not a valid route.

But there is another issue right now. Since 'fetchers' is of type 'Record<"/" | "/about" | "/contact", () => any>', it means that we must have a fetcher for each route defined in the 'routes' array. So, if we try to omit a fetcher for any route, we will also get an error. What if we want to make the fetchers optional for each route? That is, even if we don't have a fetcher for a particular route, we should not get an error. In that case, we can update our identity function to make the keys inside fetchers optional -

    const makeConfigObj = <TRoute extends string>(obj: {
        routes: TRoute[],
        fetchers: Partial<Record<TRoute, () => any>>
    }) => obj;

Here, we used the 'Partial' utility type to make the keys in the 'fetchers' object optional. Now, we can have a config object where some routes do not have corresponding fetchers, and we will not get any errors.

    const config = makeConfigObj({
        routes: ["/", "/about", "/contact"],
        fetchers: {
            "/": () => { // No error
                return {};
            }
        }
    });

In fact, even if we don't have any fetchers at all, we will still not get any errors -

    const config = makeConfigObj({
        routes: ["/", "/about", "/contact"],
        fetchers: {} // No error
    });

# WHY USE AN IDENTITY FUNCTION INSTEAD OF GENERIC TYPES?

You can also say that why even have to create an identity function for this? Why can't we just define the type of the config object directly using generic types? Something like -  

    type Config<TRoute extends string> = {
        routes: TRoute[],
        fetchers: Partial<Record<TRoute, () => any>>
    };

    const configObj: Config<"/" | "/about" | "/contact"> = {
        routes: ["/", "/about", "/contact"],
        fetchers: {
            "/does-not-exist": () => {
                return {};
            }
        }
    };

Yes, this is also a valid way to define the type of the config object, and it will work just fine. But you can see that in this case, we have to explicitly specify the type parameter 'TRoute' when defining the 'configObj'. This can be a bit cumbersome and less intuitive, especially if we have a large config object with many routes.

The benefit of identity functions is that since we use generic functions, we can infer the type parameters from the arguments passed to the function. This means that we don't have to explicitly specify the type parameters when defining the config object. The TypeScript compiler will automatically infer the types based on the values we provide.

Also, you may argue that why we have to write extra functions. Won't they affect the size of our code's bundle? Well, in most of the minifiers, they remove the identity functions during the minification process, so they won't have any impact on the final bundle size. Moreover, the benefits of having better type safety and improved developer experience often outweigh the minor increase in code size due to the identity functions.
