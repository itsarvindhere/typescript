Let's try another builder pattern example. This will help up understand how to correctly implement the builder pattern with classes and generics in TypeScript.

Let's say that we have a class -

    export class DynamicReducer {
        private handlers = {} as unknown;

        addHandler(
            type: unknown, 
            handler: (state: unknown, payload: unknown) => unknown
        ) {
            this.handlers[type] = handler;
            return this;
        }

        reduce(state: unknown, action: unknown): unknown {
            const handler = this.handlers[action.type];
            if (!handler) {
                return state;
            }

            return handler(state, action);
        }
    }

Alright. So, we have a 'DynamicReducer' class that allows us to add handlers for different action types and then reduce a state based on an action. The 'addHandler' method takes in an 'action' type and a 'handler' function, and the 'reduce' method takes in the current state and an action, and returns the new state based on the handler for that action type.

So, this is how we expect this class to be used -

    interface State {
        username: string;
        password: string
    }

    const reducer = new DynamicReducer<State>()
        .addHandler(
            "LOG_IN",
            (state, action: { username: string; password: string }) => {
                return {
                    username: action.username,
                    password: action.password
                }
            }
        )
        .addHandler("LOG_OUT", () => {
            return {
                username: "",
                password: ""
            }
        });

Here, we are creating a 'DynamicReducer' instance for a specific 'State' type. We then add two handlers - one for the "LOG_IN" action and another for the "LOG_OUT" action. The "LOG_IN" handler takes in the current state and an action with a username and password, and returns a new state with those values. The "LOG_OUT" handler simply returns an empty state.

Now, we can call the reduce method like this -

    const state1 = reducer.reduce(
        { username: "", password: ""},
        { type: "LOG_IN", username: "foo", password: "bar" }
    );

What we actually want now is this class to be type safe. We should not be allowed to say -

    const state2 = reducer.reduce(
        { username: "foo", password: "bar"},
        { type: "NOT_ALLOWED" } // This should error out
    );

Right now this won't give any error but ideally, we want to make sure that the 'reduce' method only accepts action types that have been added as handlers. In this case, the "NOT_ALLOWED" action type should not be accepted since we haven't added a handler for it. The allowed values right now for the 'type' property in the action are "LOG_IN" and "LOG_OUT". We want to enforce this at compile time using TypeScript's type system.

Moreover, we should not be allowed to say - 
    
    const state3 = reducer.reduce(
        { username: "foo", password: "bar"},
        { type: "LOG_IN" }
    );

Here, even though "LOG_IN" is a valid action type, the payload for that action type is missing. The "LOG_IN" handler expects an action with a 'username' and 'password', so we should get a compile-time error if we try to call 'reduce' with an action that doesn't have the required payload.

Basically, if the type is' 'LOG_IN', our 'reduce' method accepts a second argument which is of type - 

    { type: "LOG_IN", username: string, password: string }

Similarly, if the type is 'LOG_OUT', our 'reduce' method accepts a second argument which is of type - 

    { type: "LOG_OUT" }

So, this is all we want to achieve with generic types. Let's dive in.

# THE SOLUTION

First of all, since we are creating a new object by passing a generic type argument to the class, this means we need at least one type argument for the 'State'. So, let's add that -

    export class DynamicReducer<TState> {...}

Alright. Now, where do we actually use this 'TState' type? Well, if you think about it, all the places where we have the 'state' parameter, we can replace it with 'TState'. Because one instance of 'DynamicReducer' will be working with a specific state type, and that state type is what we are representing with 'TState'. So, let's replace all the 'state' parameters with 'TState' -

    export class DynamicReducer<TState> {
        private handlers = {} as unknown;

        addHandler(
            type: unknown,
            handler: (state: TState, payload: unknown) => TState
        ): unknown {
            this.handlers[type] = handler;

            return this;
        }

        reduce(state: TState, action: unknown): TState {
            const handler = this.handlers[action.type];
            if (!handler) {
                return state;
            }

            return handler(state, action);
        }
    }

Notice that here, we also replaced the return types of the 'handler' function and the 'reduce' method with 'TState'. This is because the handler function is supposed to take in a state of type 'TState' and return a new state of type 'TState', and the 'reduce' method is also supposed to return a new state of type 'TState'.

Alright, that's one thing done.

Now, let's think of what the 'handlers' parameter will be because right now it is typed as 'unknown'. Looking at the 'addHandlers' function, it looks like the 'handlers' object will have keys of type 'string' (the action type) and values of type '(state: TState, payload: unknown) => TState' (the handler function). So, we can type the 'handlers' parameter as -

    private handlers = {} as Record<string, (state: TState, payload: unknown) => TState>;

Now, this 'payload' is still 'unknown' here but we can make it 'any' because we know that whenever we add something to this object, we will add the right payload based on the type. So, we can change it to -

    private handlers = {} as Record<string, (state: TState, payload: any) => TState>;

Alright, let's now move on to the 'addHandler' method. Since we are following the builder pattern, we know that whatever this 'addHandler' returns should be typed such that it has the updated result after calling 'addHandler'. So, we need to keep updating that result as we keep adding new handlers.

But, what should that result look like? Well, if we look at how we are calling the 'reduce' method, we can see that the second argument to the 'reduce' method is an object with a 'type' property and some other properties based on the action type. So, we definitely need to keep track of what payloads are associated with each action type. That's what we are going to keep track of and update as we keep adding new handlers.

So, it means, we need to introduce a new generic type parameter to keep track of the action types and their associated payloads. Let's call that 'TPayloadMap'. This 'TPayloadMap' will be a mapping of action types to their payloads. So, we can define it as -

    export class DynamicReducer<TState, TPayloadMap> {...}

Now, we also want to infer the actual 'type' and the 'payloads' passed to the 'addHandler' which means the 'addHandler' itself needs to be generic.

    addHandler<TType extends string, TPayload>(
        type: TType,
        handler: (state: TState, payload: TPayload) => TState
    ) {...}

So here, we are saying that 'TType' is a generic type parameter that extends 'string' (because action types are strings), and 'TPayload' is another generic type parameter that represents the payload for that action type. The 'handler' function now takes in a payload of type 'TPayload'. This means, the inference should work perfectly when we call 'addHandler'. The only missing piece for 'addHandler' now is to return the correct type. We know that after we call 'addHandler', it will update the 'TPayloadMap' to include the new action type and its associated payload. So, we can return a new instance of 'DynamicReducer' with the updated 'TPayloadMap' -

    addHandler<TType extends string, TPayload>(
        type: TType,
        handler: (state: TState, payload: TPayload) => TState
    ): DynamicReducer<TState, TPayloadMap & Record<TType, TPayload>> {
        this.handlers[type] = handler;

        return this;
    }

So here, the return type of 'addHandler' is a new 'DynamicReducer' with the same 'TState' but an updated 'TPayloadMap' that includes the new action type and its payload. We are using an intersection type here to combine the existing 'TPayloadMap' with a new record that maps the new action type to its payload.

Now, if you hover over the 'const reducer' that we have created, you will see its type as -

    const reducer: DynamicReducer<State, Record<"LOG_IN", {
        username: string;
        password: string;
    }> & Record<"LOG_OUT", unknown>>

Hmm.... For the 'LOG_OUT' action type, the payload is 'unknown' because we haven't specified it. But since 'LOG_OUT' does not have a payload, ideally, the payload should just be an empty object. So, we can change the 'addHandler' method to allow for that as well. We can make the 'TPayload' generic type parameter optional and default it to an empty object. So, we can change the 'addHandler' signature to -

    addHandler<TType extends string, TPayload = {}>(
        type: TType,
        handler: (state: TState, payload: TPayload) => TState
    ): DynamicReducer<TState, TPayloadMap & Record<TType, TPayload>> {
        this.handlers[type] = handler;

        return this;
    }

Another error that we get is when we call 'new DynamicReducer<State>()' because right now, we are passing only one type argument but as per our class, it expects two type arguments. But we do not want to provide a argument explicitly for the 'TPayloadMap' because initially, it should start with an empty object. So, we can default it -

    export class DynamicReducer<TState, TPayloadMap = {}> {..}

Alright, so it looks like 'addHandler' is done.

Next, we have the 'reduce' function. It expects an 'action' as the second argument which is an object that not only has the 'type' but also all the properties of the 'payload' of that type.

Confusing?

Let's say that we have two actions and our TPayloadMap is -

    {
        "LOG_IN": {
            username: string;
            password: string;
        },
        "LOG_OUT": {}
    }

The 'reduce' function expects us to pass this if our 'action' type is 'LOG_IN' -    

    reduce(state, { type: "LOG_IN", username: "foo", password: "bar" })

Similarly, if our 'action' type is 'LOG_OUT', it expects us to pass this -

    reduce(state, { type: "LOG_OUT" })

So, we somehow need to get a new type out of the 'TPayloadMap' which takes each key in the 'TPayloadMap' and creates a new type that has a 'type' property with the value of that key and all the properties of the payload for that key. 
This is a perfect use case for mapped types. We can create a new type called 'PayloadsToDiscriminatedUnion' that takes a Type of Record and creates a new type based on it. We can define it like this -

    type PayloadsToDiscriminatedUnion<T extends Record<string,any>> = {
        [K in keyof T]: { type: K } & T[K]
    }[keyof T];

What is happening here is that we are iterating over each key 'K' in the 'T' and creating a new type that has a 'type' property with the value of 'K' and all the properties for that key (which is 'T[K]'). Finally, we are taking the union of all these types by indexing it with 'keyof T'

For example, if we have a 'TPayloadMap' like this -

    {
        "LOG_IN": {
            username: string;
            password: string;
        },
        "LOG_OUT": {}
    }

Then, the 'PayloadsToDiscriminatedUnion<TPayloadMap>' will evaluate to -

    { type: "LOG_IN", username: string, password: string } | { type: "LOG_OUT" }

And so, our 'reduce' function's 'action' parameter can be typed as -

    reduce(state: TState, action: PayloadsToDiscriminatedUnion<TPayloadMap>): TState {...}

But there is another error. It says -

    Type 'TPayloadMap' does not satisfy the constraint 'Record<string, any>'

This is because we have defined 'TPayloadMap' as a generic type parameter without any constraints. We need to add a constraint to 'TPayloadMap' to ensure that it is always a Record with string keys and any values. So, we can change the class definition to -

    export class DynamicReducer<TState, TPayloadMap extends Record<string, any> = {}> {...}

And that's pretty much it. We are done with making our 'DynamicReducer' class type safe. Now, if you try to call 'reduce' with an action type that is not in the 'TPayloadMap', you will get a compile-time error. Similarly, if you try to call 'reduce' with an action that does not have the required payload properties, you will also get a compile-time error.

So, our final class looks like this -

    export class DynamicReducer<TState, TPayloadMap extends Record<string,any> = {}> {
        private handlers = {} as Record<string, (state: TState, payload: any) => TState>;

        addHandler<TType extends string, TPayload = {}>(
            type: TType,
            handler: (state: TState, payload: TPayload) => TState
        ): DynamicReducer<TState, TPayloadMap & Record<TType, TPayload>> {
            this.handlers[type] = handler;

            return this;
        }

        reduce(state: TState, action: PayloadsToDiscriminatedUnion<TPayloadMap>): TState {
            const handler = this.handlers[action.type];
            if (!handler) {
                return state;
            }

            return handler(state, action);
        }
    }

Yes, at first glance, it will look complicated but if you go step by step, it is not that difficult to understand. The key takeaway here is to understand how we can use generics and mapped types to create a type-safe builder pattern with classes in TypeScript. This also shows that in Builder Pattern in TypeScript, it is important to understand what you actually want to keep track of and what type should actually be updated as you keep adding new things to the builder. In this case, we wanted to keep track of the action types and their associated payloads, and we used generics and mapped types to achieve that. 

In previous examples, we were keeping track of a type which was the same type as one of the variables in the class. But in this case, that's not the case. The 'handlers' variable has a different type than the 'TPayloadMap' that we are keeping track of. This shows that you can keep track of any type you want in the builder pattern, it doesn't have to be the same type as one of the variables in the class. The important thing is to understand what you want to keep track of and how to update it correctly as you add new things to the builder.
