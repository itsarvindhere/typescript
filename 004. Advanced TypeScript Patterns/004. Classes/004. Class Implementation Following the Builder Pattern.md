So, what is a Builder Pattern?

In software engineering, the 'Builder Pattern' is a design pattern that allows for the step-by-step construction of complex objects. It separates the construction of an object from its representation, allowing the same construction process to create different representations.

In TypeScript, we can use Classes and Generics to implement the Builder Pattern effectively. This pattern is particularly useful when you want to create an object that requires multiple steps to be constructed, or when you want to create different types of objects using the same construction process.

Let's take a very simple example.

Suppose, we want to create a class that deals with a list and has a 'push' method in it to update the list by adding an element into it. Now, we want to create the 'push' method such that we can call it like -

    <class object>.push(1).push(2).push(3)

That is, we want to be able to chain the 'push' method calls. This is where the Builder Pattern comes in handy.

So, we can do something like this (without generics) -

    export class BuilderTuple {
        list: any[];

        constructor() {
            this.list = [];
        }

        push(num: number) {
            this.list.push(num);
            return this;
        }
    }

So here, we have a class 'BuilderTuple' that has a property 'list' which is an array. The 'push' method takes a number as an argument, pushes it into the 'list', and then returns 'this', which allows us to chain the method calls.

So, we can now use this like this -

    const builderBeforePush = new BuilderTuple();
    const listBeforePush = builderBeforePush.list; // []

    const builderAfterPush = builderBeforePush.push(1).push(2).push(3);
    const listAfterPush = builderAfterPush.list; // [1, 2, 3]

While this code works perfectly on the runtime, the issue is that right now, TypeScript cannot properly infer the types of the 'list' property after the 'push' method is called.

Maybe one thing we can do is to make the 'push' method a generic method that takes a type parameter 'T' and updates the type of the 'list' property accordingly. This way, we can ensure that the types are correctly inferred throughout the chaining process.

    push<TNum extends number>(num: TNum) {
        this.list.push(num);
        return this;
    }

Here, we have made the 'push' method a generic method that takes a type parameter 'TNum' which extends 'number'. This way, we can ensure that the types are correctly inferred throughout the chaining process.

Now, if you hover over the 'push' method you will see that TypeScript will actually infer the actual number that we are pushing into the list, which is pretty cool. So, if we call 'push(1)', TypeScript will infer that 'TNum' is '1', and if we call 'push(2)', it will infer that 'TNum' is '2', and so on.

But still, there is one issue.

Since the 'list' is typed as 'any[]', TypeScript cannot infer the types of the elements in the 'list' properly. So, we need to make the 'list' property a generic property as well, and update its type accordingly when we call the 'push' method.

We can make the class itself a generic class that takes a type parameter 'TList' which represents the type of the elements in the 'list' -

    export class BuilderTuple<TList extends any[]> {
        list: TList;
        ...
    }

While this is fine, now, this piece of code starts showing an error -

    this.list = [];

The error says -

    Type 'never[]' is not assignable to type 'TList'.
        'never[]' is assignable to the constraint of type 'TList', but 'TList' could be instantiated with a different subtype of constraint 'any[]'

This is because TypeScript cannot infer the type of 'TList' at the time of class instantiation, and it defaults to 'never[]', which is not assignable to 'TList'. One way to fix it is to the 'as unknown' assertion to the empty array -

    this.list = [] as unknown as TList;

Here, we are first asserting the empty array as 'unknown', which allows us to bypass the type checking, and then we are asserting it again as 'TList', which tells TypeScript that we want to treat it as 'TList'. This way, we can avoid the error and still maintain the type safety of our class.

Now, while the errors go away, there is still one issue. If you hover over the object that we created, its type is 'BuilderTuple<any[]>'. This 'any' is not ideal because it defeats the purpose of using generics in the first place. We want to be able to infer the types of the elements in the 'list' property correctly. The reason why it has 'any' is because when we create an object, we are not providing any type argument for 'TList', so TypeScript defaults it to 'any[]'. To fix this, we can provide a default type argument for 'TList' when we define the class -

    export class BuilderTuple<TList extends any[] = []> {
        list: TList;
        ...
    }

Alright, now, let's finally change the 'push' method a bit. The reason is that we have this code -

    const builderBeforePush = new BuilderTuple();
    const listBeforePush = builderBeforePush.list; // []

    const builderAfterPush = builderBeforePush.push(1).push(2).push(3);
    const listAfterPush = builderAfterPush.list; // [1, 2, 3]

Here, if you hover over 'builderBeforePush' and 'builderAfterPush', you will see that their types are 'BuilderTuple<[]>', which is not ideal because we want 'builderBeforePush' to have the type 'BuilderTuple<[]>', and 'builderAfterPush' to have the type 'BuilderTuple<[1, 2, 3]>'.

To achieve this, we can update the 'push' method and assert the type of 'this' as -

    push<TNum extends number>(num: TNum) {
        this.list.push(num);
        return this as unknown as BuilderTuple<[...TList, TNum]>;
    }

So, here, we are asserting the type of 'this' as 'BuilderTuple<[...TList, TNum]>', which tells TypeScript that after calling the 'push' method, the type of 'this' will be a 'BuilderTuple' with a 'list' that has all the previous elements of 'TList' plus the new element 'TNum'. This way, we can ensure that the types are correctly inferred throughout the chaining process.

And the wonderful thing about all this code is that now, if you hover over 'builderBeforePush', you will see that its type is 'BuilderTuple<[]>', and if you hover over 'builderAfterPush', you will see that its type is 'BuilderTuple<[1, 2, 3]>', which is exactly what we wanted.

This way, we have successfully implemented the Builder Pattern in TypeScript while maintaining strong type safety and accurate type inference throughout the chaining process.

So, as you keep pushing elements into the list, TypeScript will keep inferring the types of the elements in the list correctly, and you can be confident that your code is type-safe and free from type-related errors.

And in the same manner, we can actually create a pop() method that allows us to remove the last element from the list and update the type of the 'list' property accordingly -

    pop() {
        this.list.pop();
        return this as unknown as BuilderTuple<TList extends [...infer Rest, any] ? Rest : never>;
    }

Here, we are using a conditional type to infer the type of the remaining elements in the 'list' after popping the last element. If 'TList' can be inferred as an array with some elements followed by any type (which represents the last element), we extract the 'Rest' of the elements and return a new 'BuilderTuple' with that type. If 'TList' cannot be inferred in that way, we return 'never', which indicates that the operation is not valid.

Similarly, we can maybe also create an 'unshift' method -

    unshift<TNum extends number>(num: TNum) {
        this.list.unshift(num);
        return this as unknown as BuilderTuple<[TNum, ...TList]>;
    }

Here, we are adding a new element at the beginning of the 'list' and updating the type of 'this' to reflect that change. We are using a tuple type to represent the new type of the 'list', where the new element 'TNum' is added at the beginning, followed by all the previous elements of 'TList'.

So, this is how we can use Classes and Generics in TypeScript to implement the Builder Pattern effectively, allowing us to create complex objects with a fluent interface while maintaining strong type safety and accurate type inference throughout the process.

The most beautiful thing here is how our Types also update dynamically as we call the 'push' and 'pop' methods, giving us a clear and accurate representation of the state of our object at any given time.

So, this is what the final code will look like -

    export class BuilderTuple<TList extends any[] = []> {
        list: TList;

        constructor() {
            this.list = [] as unknown as TList;
        }

        push<TNum extends number>(num: TNum) {
            this.list.push(num);
            return this as unknown as BuilderTuple<[...TList, TNum]>;
        }

        unshift<TNum extends number>(num: TNum) {
            this.list.unshift(num);
            return this as unknown as BuilderTuple<[TNum, ...TList]>;
        }

        pop() {
            this.list.pop();
            return this as unknown as BuilderTuple<TList extends [...infer Rest, any] ? Rest : never>;
        }
    }
