Let's take our knowledge of generic classes and builder pattern and apply it to one example.

Let's say we have this piece of code -

    type Middleware<TInput, TOutput> = (input: TInput) => TOutput | Promise<TOutput>;

    class DynamicMiddleware<TInput, TOutput> {
        private middleware: Middleware<any, any>[] = [];

        constructor(firstMiddleware: Middleware<TInput, TOutput>) {
            this.middleware.push(firstMiddleware);
        }

        use(middleware: Middleware<TInput, TOutput>){
            this.middleware.push(middleware);
            return this as any;
        }

        async run(input: TInput): Promise<TOutput> {
            let result: TOutput = input as any;

            for (const middleware of this.middleware) {
                result = await middleware(result);
            }

            return result;
        }
    }


So, we have a 'DynamicMiddleware' class that takes in an initial middleware function and allows us to chain additional middleware functions using the 'use' method. The 'run' method executes all the middleware functions in sequence, passing the result of each middleware to the next one.

So, we can use it like this -

    const middleware = new DynamicMiddleware((req: Request) => {
        return {
            ...req,
            userId: req.url.split("/")[2]
        };
    })
        .use((req) => {
            if (req.userId === "123") {
                throw new Error();
            }

            return req;
        })
        .use (async (req) => {
            return {
                ...req,
                user: await fetchUser(req.userId)
            };
        });

Here, we create a new instance of 'DynamicMiddleware' with an initial middleware function that extracts the user ID from the request URL. We then chain two additional middleware functions: one that checks if the user ID is "123" and throws an error if it is, and another that fetches the user data based on the user ID.

And we can use the 'run' function to provide the initial input and execute the middleware chain -

    const result = await middleware.run({
        url: "/user/matt"
    } as Request);

This is a quite common pattern in web frameworks, where you have a series of middleware functions that process a request and produce a response. By using the builder pattern and generics, we can create a flexible and reusable middleware system that can handle various types of input and output.

But there is one problem right now.

So far, our code is not error-free. We have some compile-time errors, especially in the 'use' method. In the first 'use' method that we use, we get this error -

    Argument of type '(req: Request) => Request' is not assignable to parameter of type 'Middleware<Request, { userId: string; cache: RequestCache; credentials: RequestCredentials; destination: RequestDestination; ... 18 more ...; text(): Promise<...>; }>'.

So, let's analyze it.

In the first 'use' method, we call it like this -

    .use((req) => {
        if (req.userId === "123") {
            throw new Error();
        }

        return req;
    })

So, its argument is this function -

    (req) => {
        if (req.userId === "123") {
            throw new Error();
        }

        return req;
    }

Looking at this, we can say that its type is -

    '(req: Request) => Request'

If we look at the 'use' method, we have specified its input parameter to be of type -

    Middleware<TInput, TOutput>

Here, 'TInput' is basically the input we provided to the very first middleware function when we created the 'DynamicMiddleware' instance. In our case, it is the 'Request' type. So, we can say that 'TInput' is 'Request'.

And 'TOutput' is the output of the very first middleware function. In our case, it is this type -

    {
        userId: string;
        cache: RequestCache;
        credentials: RequestCredentials;
        destination: RequestDestination;
        ... 18 more ...;
        text(): Promise<...>;
    }

This is where the problem occurs. Our 'use' method outputs a type of 'Request', but it is not assignable to the output type of the first middleware function. Hence, we get this error. But now the question is, how do we fix it?

If we think for a minute. Whatever the 'use' method accepts should be a function whose 'input' is the 'output' of previous middleware function.

That is, in case of our first 'use' method, its input should be -

    {
        userId: string;
        cache: RequestCache;
        credentials: RequestCredentials;
        destination: RequestDestination;
        ... 18 more ...;
        text(): Promise<...>;
    }

And the output is something we do not know ahead of time but we can make TypeScript infer it using Generics. So, our first task is to make the 'use' method generic so that we can infer its output -

    use<TNewOutput>(middleware: Middleware<TOutput, TNewOutput>){
        this.middleware.push(middleware);
        return this as any;
    }

Here, we have made the 'use' method generic by introducing a new type parameter 'TNewOutput'. This allows us to infer the output type of the middleware function passed to the 'use' method based on the output type of the previous middleware function.

Notice how now, the middleware that we accept in the 'use' method has its input type as 'TOutput', which is the output type of the previous middleware function. This way, we ensure that the middleware functions are chained correctly, and the types are consistent throughout the chain.

So, the error goes away from the first 'use' method but now we have another error. In the second 'use' method we have -

    .use (async (req) => {
        return {
            ...req,
            user: await fetchUser(req.userId)
        };
    });

Here, we are getting an error for the 'req' parameter saying -

    Parameter 'req' implicitly has an 'any' type.

But why?

Well, the answer is pretty simple. In the 'use' method, we return 'this as any'. So, it means the first time we use the 'use' method, it returns 'any'.

So, afterwards, whatever we do with the returned value from the 'use' method, we do that on the 'any' type. So, we lose all the type information and hence, we get this error.

So, somehow, we also need to correctly type the 'return type' of the 'use' method so that we can maintain the type information throughout the chain.

We know that this method returns 'this' which means an instance of the 'DynamicMiddleware' class. So, the return type should be of type 'DynamicMiddleware' but the thing is, we need to properly specify the type parameters of the 'DynamicMiddleware' class in the return type of the 'use' method.

If we think for a bit, the 'use' method basically returns a new instance of 'DynamicMiddleware' with the same 'TInput' but a new 'TOutput' which is the output type of the middleware function passed to the 'use' method.

Why the 'TInput' remained the same? Because it never changes inside the 'use' method. It is always the input type of the very first middleware function. But the 'TOutput' changes because it is the output type of the middleware function passed to the 'use' method. If you look at the class definition, the only place where 'TInput' is used is in the constructor where we push the first middleware function.

After that, it is only used in 'run' method and there too, it is only used as the input type of the 'run' method which is in turn the input type of the first middleware function. So, it is clear that 'TInput' never changes in the 'use' method. But 'TOutput' changes because it is the output type of the middleware function passed to the 'use' method.

So, we can say -

    use<TNewOutput>(middleware: Middleware<TOutput, TNewOutput>)
        : DynamicMiddleware<TInput, TNewOutput>{
        this.middleware.push(middleware);
        return this as any;
    }

And that's it! This fixes the issue.

    const result = await middleware.run({
        url: "/user/matt"
    } as Request);

Moreover, if you now hover over the 'result', you will see that it is also correctly typed as the output of the last middleware function in the chain. This way, we have successfully built a chainable middleware system using the builder pattern and generics in TypeScript, while maintaining type safety throughout the chain.

So, when working with builder pattern in Typescript, always make sure you properly type the return type of the methods that are used for chaining, so that you can maintain the type information throughout the chain and avoid any type-related issues.
