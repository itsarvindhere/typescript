In the last section, we learned how we can use Generics and Classes in TypeScript to create beautiful type safe builder patterns. Let's take that knowledge and try to apply it to create a type-safe map.

As a starting point, let's say we have this class - 

    class TypeSafeStringMap<TMap extends Record<string, string> = {}> {
        private map: TMap;

        constructor() {
            this.map = {} as TMap;
        }

        get(key: keyof TMap): string {
            return this.map[key];
        }

        set<K extends string>(key: K, value: string):unknown {
            (this.map[key] as unknown) = value;
            return this;
        }
    }

Here, we have a class 'TypeSafeStringMap' that takes a generic type parameter 'TMap', which extends 'Record<string, string>'. This means that 'TMap' is an object type where the keys are strings and the values are also strings.

We have a private property 'map' of type 'TMap', which is initialized as an empty object in the constructor. The 'get' method takes a key of type 'keyof TMap' and returns the corresponding value from the map. The 'set' method takes a key of type 'K' (which extends string) and a value of type string, and updates the map with the new key-value pair.

The issue right now is that when we do -

    const map = new TypeSafeStringMap()
    .set("Arvind", "Rana")
    .set("John", "Cena")
    .set("James", "Rodriguez")

We get an error because 'set' is returning 'unknown', and we cannot chain the 'set' method calls.

Also, if we call the 'get' method, there is no type safety, and we can pass any string as the key, which may not exist in the map.

To fix this, we need to ensure that the 'set' method returns the correct type of the class instance, which allows for method chaining.

Now, we know that with each subsequent 'set' calls, our map will keep growing. So, we need to type the returned 'this' in such a way that it reflects the updated type of the map after each 'set' call. In case of arrays, we could use the spread operator to create a new array with the new element added, but since we are working with objects, we can use the intersection type to combine the existing map type with the new key-value pair.

If you do not know, intersection types in TypeScript allow us to combine multiple types into one. When we use an intersection type, the resulting type has all the properties of the combined types.

So, if we want to combine two types, we can use the '&' operator to create a new type that includes all the properties of both types. For example, if we have two types 'A' and 'B', we can create a new type 'C' that is the intersection of 'A' and 'B' like this:

    type A = { name: string };
    type B = { age: number };
    type C = A & B; // C will have both 'name' and 'age' properties

This is what we can use here as well.

Since we know that 'TMap' is the type of our current map, and we want to add a new key-value pair to it, we can create a new type that is the intersection of 'TMap' and an object type that has the new key and value. This way, we can ensure that the returned type from the 'set' method reflects the updated map with the new key-value pair.

    set<K extends string>(key: K, value: string): TypeSafeStringMap<TMap & Record<K, string>> {
        (this.map[key] as unknown) = value;
        return this;
    }

So here, we are returning a new instance of 'TypeSafeStringMap' with the type parameter 'TMap & Record<K, string>'. This means that the returned type will include all the properties of the existing map (TMap) as well as the new key-value pair (Record<K, string>).

The benefit of doing this is that in our code where we create a new map and set some key value pairs, TypeScript can actually infer the type of the map as we keep adding new key-value pairs. So, if we do -

    const map = new TypeSafeStringMap()
    .set("Arvind", "Rana")
    .set("John", "Cena")
    .set("James", "Rodriguez")

And nw, if you hover over the 'map' variable, you will see that TypeScript has inferred the type of 'map' as -

    const map: TypeSafeStringMap<Record<"Arvind", string> & Record<"John", string> & Record<"James", string>>

This is the beauty of builder pattern with TypeScript. It allows us to create complex types that evolve as we build our object, and TypeScript can keep track of these changes to provide us with accurate type information throughout the process.

Another benefit is that when we use 'get' method, TypeScript knows what all keys are available in the map and what their corresponding value types are. So, if we do -

    map.get(...);

TypeScript will provide us with autocomplete suggestions for the keys that we have added to the map. We cannot just randomly put any key in the 'get' method, it has to be one of the keys that we have added to the map. This ensures type safety and prevents us from making mistakes when accessing values from the map.
