Here is another cool pattern that we can have with branded types. We can use them to create more specific types for our entity ids.

Let's say we have a code where we have two interfaces -

    interface User {
        id: string;
        name: string;
    }

    interface Post {
        id: string;
        title: string;
        content: string;
    }

Here, we have two entities - User and Post. Both of them have an 'id' property which is of type string.

Let's say we have two functions - getUserById and getPostById. Both of them take an id as a parameter and return the corresponding entity.

    const db: { users: User[]; posts: Post[] } = {
        users: [
            {
                id: '1',
                name: 'Alice'
            }
        ],
        posts: [
            {
                id: '1',
                title: 'Hello World',
                content: 'This is my first post'
            }
        ]
    };

    const getUserById = (id: string) => {
        return db.users.find(user => user.id === id);
    }

    const getPostById = (id: string) => {
        return db.posts.find(post => post.id === id);
    }

At first glance, it seems like there is nothing wrong. And yes, there won't be any compile time errors. But the issue with this code is that we can easily mix up the ids. We can pass a post id to the 'getUserById' function and it will return 'undefined'. Similarly, we can pass a user id to the 'getPostById' function and it will also return 'undefined'.

    const postId = "123";
    const userId = "1";

    // No ERRORS shown by TypeScript but it will return undefined at runtime
    const user = getUserById(postId); // Passing post id to getUserById
    const post = getPostById(userId); // Passing user id to getPostById

What if we want to ensure that the 'getUserById' should only get a user id and the 'getPostById' should only get a post id? This is where branded types come in handy.

We can create two branded types - UserId and PostId. These types will be based on the string type but they will be branded with a unique symbol.

    type UserId = Branded<string, 'UserId'>;
    type PostId = Branded<string, 'PostId'>;

And now, our functions should take these branded types as parameters instead of just string.

    const getUserById = (id: UserId) => {
        return db.users.find(user => user.id === id);
    }

    const getPostById = (id: PostId) => {
        return db.posts.find(post => post.id === id);
    }

Now, this code will show an error -

    const postId = "123";
    const userId = "1";

    const user = getUserById(postId);
    const post = getPostById(userId);

The reason why it shows an error is because we are passing normal string values to the functions which are expecting branded types. To fix this, we need to cast the string values to the corresponding branded types.

    const postId = "123" as PostId;
    const userId = "1" as UserId;

    const user = getUserById(postId);
    const post = getPostById(userId);

And now, we can another error because now, we are passing a post id to the 'getUserById' function and a user id to the 'getPostById' function.

    const postId = "123" as PostId;
    const userId = "1" as UserId;

    // Argument of type 'PostId' is not assignable to parameter of type 'UserId'.
    const user = getUserById(postId); 

    // Argument of type 'UserId' is not assignable to parameter of type 'PostId'.
    const post = getPostById(userId);

This is so wonderful because now, we can catch these kinds of errors at compile time instead of runtime. We can ensure that the 'getUserById' function only gets a user id and the 'getPostById' function only gets a post id. This is one of the main advantages of using branded types.

But we are not done yet. If you notice, our interfaces are still using 'string' type for the 'id' property. We can also change that to use the branded types.

    interface User {
        id: UserId;
        name: string;
    }

    interface Post {
        id: PostId;
        title: string;
        content: string;
    }

    const db: { users: User[]; posts: Post[] } = {
        users: [
            {
                id: '1' as UserId,
                name: 'Alice'
            }
        ],
        posts: [
            {
                id: '123' as PostId,
                title: 'Hello World',
                content: 'This is my first post'
            }
        ]
    };

This ensures that wherever we are using the 'id' property of the User or Post interface, we are using the branded types. This way, we can ensure that we are not mixing up the ids anywhere in our code.

Yes, it means you will have to explicitly 'assert' the string values to the corresponding branded types whenever you are creating a new user or post. But this is a small price to pay for the safety and correctness that we get from using branded types.
    
This is a really powerful pattern that we can use to create more specific types for our entity ids and ensure that we are not mixing them up anywhere in our code. It helps us catch errors at compile time and makes our code more robust and maintainable.
