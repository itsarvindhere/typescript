Let's look at a practical example of using branded types for form validation. Suppose we have a form where users can enter their email address and password and before we use these to authenticate the user, we want to ensure that the email and password meet certain criteria.

    type Password = Branded<string, 'Password'>;
    type Email = Branded<string, 'Email'>;

    const validateValues = (values: { email: string; password: string }) => {
        if (!values.email.includes('@')) {
            throw new Error('Invalid email');
        }

        if (values.password.length < 8) {
            throw new Error('Password must be at least 8 characters long');
        }

        return {
            email: values.email,
            password: values.password
        }
    }


So here, we have a 'validateValues' function that checks if the email contains an '@' symbol and if the password is at least 8 characters long. If either of these conditions is not met, an error is thrown.

Now, we have a method that creates a new user in the database if the validation passes -

    const createUserOnApi = (values: {email: Email; password: Password}) => {
        // ...
    };

Notice that this method takes parameters of type 'Email' and 'Password', which are branded types. This means that we can only call 'createUserOnApi' with values that have been validated and branded as 'Email' and 'Password'.

Let's say when form is submitted, this method gets called -

    const onSubmitHandler = (values: { email: string; password: string}) => {
        const validatedValues = validateValues(values);

        createUserOnApi(validatedValues);
    }

Here, the 'onSubmitHandler' function takes the raw form values, validates them using 'validateValues', and then passes the validated values to 'createUserOnApi'.

The issue right now is that we will get an error when we try to pass 'validatedValues' to 'createUserOnApi' because 'validatedValues' is of type '{ email: string; password: string }', while 'createUserOnApi' expects parameters of type '{ email: Email; password: Password }'.

So, how do we fix this?

Well, we know that since 'createUserOnApi' expects branded types, we need to brand the validated values before passing them to 'createUserOnApi'. We can do this by using type assertions to assert that the validated values are of the correct branded types.

Inside the 'validateValues' function, after we have validated the email and password, we can return the values as branded types like this -

    const validateValues = (values: { email: string; password: string }) => {
        if (!values.email.includes('@')) {
            throw new Error('Invalid email');
        }

        if (values.password.length < 8) {
            throw new Error('Password must be at least 8 characters long');
        }

        return {
            email: values.email as Email,
            password: values.password as Password
        }
    }

So now, the error will be resolved because we have asserted that the validated values are of the correct branded types. This way, we can ensure that only valid email and password values are passed to 'createUserOnApi', providing a safer and more robust form validation mechanism.

What's the benefit of using branded types in this scenario?

The benefit of using branded types in this scenario is that it provides a clear and type-safe way to ensure that only valid data is passed to the 'createUserOnApi' function. By using branded types, we can enforce that the email and password have been validated before they are used for authentication. If we do not get the correct branded types, we will get a compile-time error, which helps catch potential issues early in the development process.

This helps prevent potential bugs and security issues that could arise from using invalid data. Additionally, it makes the code more readable and maintainable by clearly indicating the expected types of the parameters.
